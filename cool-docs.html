Cool Language
Documentation

A native-first, high-level systems language with Python-like syntax, a bytecode VM, LLVM native compiler, FFI, and freestanding build support.

Get Started View on GitHub

Introduction

Cool is a high-level systems language with Python-like syntax and a native-first toolchain. Its primary compilation path targets native binaries and freestanding object files through LLVM, while the interpreter and bytecode VM provide fast iteration, tooling support, and runtime parity. It also ships with FFI, ABI and data-layout controls, and bundled terminal apps written in Cool itself.

Indentation Blocks
Python-style. No braces, no end keywords.
Classes & OOP
Inheritance, super(), operator overloading.
Closures
Functions capture their scope. nonlocal & global supported.
f-strings
Interpolation, expressions, method calls inline.
Standard Library
math, os, sys, json, re, time, random, collections, path, csv, datetime, hashlib, platform, toml, yaml, sqlite, http, subprocess, argparse, logging, test, socket, term, ffi, and more.
Native Builds
LLVM backend, native binaries, freestanding object output, and ABI-level control.
Rust Implementation
Interpreter, VM, native compiler, and tooling implemented in Rust.
MIT Licensed
Free to use, modify, and distribute.

Getting Started

Prerequisites

You need Rust (stable, edition 2021) installed. LLVM 17 is only required for native compilation (cool build or cool --compile).

Build from Source

bash
git clone https://github.com/codenz92/cool-lang
cd cool-lang
cargo build --release

The binary will be at ./target/release/cool.

Running Cool

Command Description
./target/release/cool build hello.cool Compile a source file to a native binary (requires LLVM 17)
./hello Run the compiled native binary
./target/release/cool build --freestanding hello.cool Emit a freestanding object file (hello.o)
./target/release/cool hello.cool Run a source file with the tree-walk interpreter
./target/release/cool --vm hello.cool Run a source file with the bytecode VM
./target/release/cool Start the interactive REPL
./target/release/cool apps/shell.cool Launch the bundled Cool shell app
./target/release/cool help Show the full CLI help

Project Workflow

bash
cool new myapp
cd myapp

cool test
cool add toolkit --path ../toolkit
cool add theme --git https://github.com/acme/theme.git
cool install
cool task list
cool build
cool build --freestanding

Manifest-driven projects use cool.toml with [project], [dependencies], and [tasks]. cool add updates the manifest, cool install materializes git dependencies into .cool/deps, cool test discovers test_*.cool / *_test.cool, and cool task runs manifest-defined workflows.

Closures

cool
def make_counter(start=0):
    count = start
    def increment(by=1):
        nonlocal count
        count += by
        return count
    return increment

c(5)    # 16

nonlocal / global

cool
count = 0

def increment():
    global count     # access module-level variable

def outer():
    x = 0
    def inner():
        nonlocal x   # access enclosing function's variable
        x += 1
    inner()
    return x         # 1

cool
class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)

    def __eq__(self, other):
        return self.x == other.x and self.y == other.y

    def __len__(self):
        return 2

    def __str__(self):
        return f"({self.x}, {self.y})"

v1 = Vector(1, 2)
v2 = Vector(3, 4)
print(v1 + v2)     # (4, 6)
print(v1 * 3)      # (3, 6)
print(len(v1))     # 2

Context Managers (with)

Custom __enter__ / __exit__ context managers work in both the tree-walk interpreter and the bytecode VM, including cleanup on normal exit, control-flow exits, and raised exceptions. LLVM native builds now support custom context managers and with open(...) on normal exit, control-flow exits, caught exceptions, and unhandled native raises too.

cool
class Timer:
    def __enter__(self):
        self.start = time.monotonic()
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        elapsed = time.monotonic() - self.start
        print(f"Elapsed: {elapsed:.4f}s")
        return false   # don't suppress exceptions

with Timer():
    do_heavy_work()

Structs

Structs are value types with named, typed fields. They work in all three runtimes (interpreter, VM, and LLVM native builds) and produce stable binary layouts when compiled.

cool
struct Point:
    x: i32
    y: i32

p = Point(10, 20)          # positional
q = Point(x=3, y=4)        # keyword

print(p.x, p.y)            # 10 20
p.x = 99

Use packed struct for consecutive byte layout with no inter-field padding — useful for binary protocols, file formats, and systems work:

cool
packed struct Header:
    magic: u16
    version: u8
    length: u32

h = Header(magic=0xCAFE, version=1, length=256)
print(h.magic, h.version, h.length)
Field types: i8, u8, i16, u16, i32, u32, i64, u64, f32, f64, bool. The LLVM backend uses real LLVM struct types and GEP-based field access for stable ABI.

Error Handling

try / except / else / finally

cool
try:
    result = int(user_input)
    value  = 100 / result
except ZeroDivisionError:
    print("cannot divide by zero")
except ValueError as e:
    print(f"not a valid number: {e}")
except Exception as e:
    print(f"unexpected error: {e}")
else:
    print(f"result = {value}")   # runs only if no exception
finally:
    print("always runs — cleanup here")

raise

cool
def divide(a, b):
    if b == 0:
        raise ValueError("divisor cannot be zero")
    if type(a) != int or type(b) != int:
        raise TypeError("arguments must be integers")
    return a / b

Exception Propagation

Uncaught exceptions propagate up the call stack automatically:

cool
def parse(data):
    return int(data["value"])   # may raise ValueError

def process(raw):
    return parse(raw)           # exception propagates up

try:
    process({"value": "bad"})
except ValueError as e:
    print(f"caught it: {e}")

ffi (Foreign Function Interface)

Call native C functions from shared libraries. This enables low-level systems programming and access to native libraries.

cool
import ffi

# Open a shared library
libm = ffi.open("libm")           # math library on Unix
libc = ffi.open("libc")          # C standard library

# Define a function: ffi.func(lib, name, return_type, [arg_types])
sqrt_fn = ffi.func(libm, "sqrt", "f64", ["f64"])
abs_fn  = ffi.func(libc, "abs",  "i32", ["i32"])

# Call the native function
sqrt_fn(2.0)    # 1.414213...
abs_fn(-42)     # 42

with Statement (Recommended)

Use with for structured file cleanup. Both the tree-walk interpreter and the bytecode VM close files on normal exit and raised exceptions. LLVM native builds now support open() and file context managers on normal exit, control-flow exits, caught exceptions, and unhandled native raises too.

cool
import time

# Read entire file
with open("data.txt", "r") as f:
    content = f.read()

# Read lines
with open("data.txt", "r") as f:
    lines = f.readlines()
    for line in lines:
        print(line.strip())

# Write
with open("output.txt", "w") as f:
    f.write("Hello, file!\n")
    f.write("Second line\n")

# Append
with open("log.txt", "a") as f:
    f.write(f"[{time.time()}] Event occurred\n")

How the Interpreter Works

Cool uses a classic pipeline architecture:

  1. Lexer (lexer.rs) — scans source text into tokens. Handles Python-style indentation by emitting synthetic INDENT and DEDENT tokens.
  2. Parser (parser.rs) — recursive descent parser that consumes the token stream and produces an Abstract Syntax Tree (AST).
  3. Interpreter (interpreter.rs) — walks the AST and evaluates each node. Manages scope, closures, classes, and all built-in operations.

Phase 8 adds a compiler backend alongside the interpreter:

  1. Compiler (compiler.rs) — translates the AST into a flat bytecode instruction sequence.
  2. VM (vm.rs) — stack-based virtual machine that executes bytecode. Defined opcodes live in opcode.rs.
